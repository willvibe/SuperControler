## 客户端需要的服务端接口信息

基于当前服务端的完整实现，客户端（主控端和被控端）需要了解以下接口：

---

### 一、WebSocket 连接

| 项目     | 说明                         |
| -------- | ---------------------------- |
| **地址** | `wss://launch.cc.cd:8765/ws` |
| **协议** | WebSocket over TLS (WSS)     |

---

### 二、消息类型（客户端需要发送的）

#### 1. 注册设备（被控端 & 主控端都需要）
```json
{
  "type": "register",
  "payload": {
    "device_id": "唯一设备标识",
    "public_key": "设备的公钥（用于加密通信）",
    "udp_port": 12345
  }
}
```

#### 2. 请求在线设备列表（主控端）
```json
{
  "type": "get_devices"
}
```

#### 3. 连接目标设备（主控端发起控制）
```json
{
  "type": "connect",
  "payload": {
    "target_id": "要控制的设备ID",
    "public_key": "主控端公钥",
    "udp_port": 主控端UDP端口
  }
}
```

#### 4. P2P 打洞失败通知（双方）
```json
{
  "type": "p2p_fail"
}
```

#### 5. 中继消息（P2P 失败后的数据转发）
```json
{
  "type": "relay",
  "payload": { /* 任意数据 */ }
}
```

---

### 三、消息类型（客户端需要接收处理的）

| 消息类型         | 触发场景                 | Payload 内容                                                |
| ---------------- | ------------------------ | ----------------------------------------------------------- |
| `registered`     | 注册成功后               | `{"public_ip": "服务器检测到的公网IP"}`                     |
| `devices_list`   | 发送 `get_devices` 后    | `{"count": N, "devices": [...]}`                            |
| `device_online`  | 其他设备上线时广播       | `{"device_id": "...", "public_ip": "...", "udp_port": ...}` |
| `device_offline` | 其他设备下线时广播       | `{"device_id": "..."}`                                      |
| `punch_info`     | `connect` 成功后双方收到 | 对方的 IP、端口、公钥、角色、会话密钥                       |
| `relay`          | P2P 失败后的数据转发     | 对方发送的原始数据                                          |
| `error`          | 操作出错时               | `{"message": "错误描述"}`                                   |

---

### 四、HTTP 接口（可选，用于调试或轮询）

| 接口                                    | 说明                          |
| --------------------------------------- | ----------------------------- |
| `GET https://launch.cc.cd:8765/health`  | 健康检查                      |
| `GET https://launch.cc.cd:8765/devices` | 获取在线设备列表（HTTP 方式） |

---

### 五、主控端实现流程

```
1. 连接 WSS → 发送 register 注册自己
2. 发送 get_devices 获取当前在线设备列表
3. 监听 device_online / device_offline 实时更新列表
4. 用户选择设备 → 发送 connect 请求
5. 收到 punch_info → 获取对方 IP/端口/密钥
6. 尝试 UDP P2P 直连
7. 如果 P2P 失败 → 发送 p2p_fail → 进入 relay 模式
```

### 六、被控端实现流程

```
1. 连接 WSS → 发送 register 注册自己
2. 保持连接，等待 punch_info（主控端发起连接时收到）
3. 收到 punch_info → 获取对方 IP/端口/密钥
4. 尝试 UDP P2P 直连
5. 如果 P2P 失败 → 发送 p2p_fail → 进入 relay 模式
```

---

### 七、关键数据结构

**PunchInfoPayload（打洞信息）：**
```json
{
  "peer_ip": "对方公网IP",
  "peer_port": 对方UDP端口,
  "peer_pubkey": "对方公钥",
  "role": "controller" | "controlled",
  "session_key": "会话密钥（用于加密）"
}
```

**DeviceInfo（设备信息）：**
```json
{
  "device_id": "设备ID",
  "public_ip": "公网IP",
  "udp_port": UDP端口,
  "peer_id": "当前连接的对端ID（如果有）"
}
```
