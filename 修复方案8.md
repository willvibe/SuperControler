### 🚨 一、 严重/致命 Bug (Critical Bugs)

#### 1. 隐藏 API 限制导致 Root 触控失效 (Android 9+)

**文件:** `core/input-server/src/main/java/com/yourapp/remotectrl/input/Main.java`

**问题分析:** `Main.java` 作为被注入到底层的 `.jar`，通过 `app_process` 运行。代码中大量使用了反射来获取 `android.hardware.input.InputManager` 和 `injectInputEvent`。从 Android 9 (API 28) 开始，系统严格限制了对非 SDK 接口（Hidden API）的反射调用。即使是在 Root (UID 0) 环境下，Dalvik/ART 虚拟机的反射限制依然生效，这会导致直接抛出 `NoSuchMethodException`，导致所有的点击、滑动指令全部无效。

**修复方案:**

必须在 `Main.java` 的入口处利用反射修改 `VMRuntime`，豁免隐藏 API 检查。

Java

```
// 在 Main.java 的 main 方法最前面添加隐藏 API 豁免
public static void main(String[] args) {
    if (args.length < 1) {
        System.err.println("Missing auth token.");
        return;
    }
    
    // 【修复新增】绕过 Android 9+ 的 Hidden API 限制
    try {
        Method getRuntimeMethod = Class.forName("dalvik.system.VMRuntime").getDeclaredMethod("getRuntime");
        Object vmRuntime = getRuntimeMethod.invoke(null);
        Method setHiddenApiExemptionsMethod = vmRuntime.getClass().getDeclaredMethod("setHiddenApiExemptions", String[].class);
        setHiddenApiExemptionsMethod.invoke(vmRuntime, new Object[]{new String[]{"L"}});
    } catch (Exception e) {
        e.printStackTrace(); // 忽略旧版本不存在此方法的异常
    }

    String authToken = args[0];
    initReflection();
    // ... 后续代码保持不变
```

#### 2. 主线程执行 Socket IO 导致崩溃 (NetworkOnMainThreadException)

**文件:** `ControlledService.kt`

**问题分析:** 在处理滚轮事件 `batchScroll` 时，代码为了将零散的滚动事件合并，使用 `Handler(Looper.getMainLooper())` 延迟发送。然而，当 Runnable 执行时，调用了 `inj.scroll(...)`，最终会调用 `InputInjector` 的 `outStream?.write(...)` 进行 LocalSocket 通讯。在 Android 主线程执行 Socket 写入，会直接引发 `NetworkOnMainThreadException` 导致 Service 崩溃。

**修复方案:** 将批处理逻辑移至协程的后台线程（IO 线程）中执行。

Kotlin

```
// ControlledService.kt 中的 batchScroll 修复
private fun batchScroll(inj: InputInjector, x: Int, y: Int, dx: Int, dy: Int) {
    if (!isBatchingScroll) {
        isBatchingScroll = true
        scrollStartX = x
        scrollStartY = y
        pendingScrollDx = dx
        pendingScrollDy = dy

        // 【修复】改用 serviceScope 和 Dispatchers.IO
        serviceScope.launch(Dispatchers.IO) {
            delay(80)
            if (pendingScrollDx != 0 || pendingScrollDy != 0) {
                val endX = scrollStartX + pendingScrollDx
                val endY = scrollStartY + pendingScrollDy
                inj.scroll(scrollStartX, scrollStartY, endX - scrollStartX, endY - scrollStartY)
            }
            isBatchingScroll = false
            pendingScrollDx = 0
            pendingScrollDy = 0
        }
    } else {
        pendingScrollDx += dx
        pendingScrollDy += dy
    }
}
```

#### 3. WebRTC 本地 ICE 候选者丢失 (导致 P2P 穿透失败)

**文件:** `WebRtcClient.kt`

**问题分析:** 在被控端初始化 WebRTC 时，`_peerId` 是在收到信令服务器的连接通知后才设置的。如果 WebRTC 底层的 ICE 收集工作在 `_peerId` 赋值之前就已经触发，代码中：

Kotlin

```
if (currentPeerId.isNotEmpty()) { 
    // 发送候选者 
} else { 
    Log.w(TAG, "onIceCandidate: peerId is empty...") 
}
```

这会导致**早期的 Local ICE Candidate 直接被丢弃**，控制端无法获取到这些 IP 端口信息，极大降低打洞成功率，甚至导致连接建立失败。

**修复方案:**

增加本地 ICE 队列，等 `peerId` 赋值后再统一发送。

Kotlin

```
// WebRtcClient.kt 新增变量
private val queuedLocalIceCandidates = CopyOnWriteArrayList<IceCandidate>()

var peerId: String
    get() = _peerId
    set(value) {
        if (_peerId.isEmpty()) {
            _peerId = value
            Log.i(TAG, "peerId set to: $value")
            flushQueuedIceCandidates() // 刷新远端 ICE
            // 【修复】立刻发送积压的本地 ICE
            queuedLocalIceCandidates.forEach { candidate ->
                signalingListener.sendIceCandidate(value, candidate.sdpMid, candidate.sdpMLineIndex, candidate.sdp)
            }
            queuedLocalIceCandidates.clear()
        }
    }

// 并在 onIceCandidate 回调中修改：
override fun onIceCandidate(candidate: IceCandidate?) {
    if (candidate == null) return
    val currentPeerId = _peerId
    if (currentPeerId.isNotEmpty()) {
        signalingListener.sendIceCandidate(currentPeerId, candidate.sdpMid, candidate.sdpMLineIndex, candidate.sdp)
    } else {
        Log.w(TAG, "onIceCandidate: peerId is empty, queuing local candidate")
        queuedLocalIceCandidates.add(candidate) // 【修复】加入队列而不是丢弃
    }
}
```

#### 4. Android 14+ MediaProjection 权限复用崩溃

**文件:** `ControlledService.kt`

**问题分析:**

Android 14 (API 34) 强化了屏幕录制隐私，`MediaProjection` 的 Intent 只能被消费**一次**。代码中 `MediaProjectionHelper.hasCachedPermission()` 会缓存旧的 Intent，并在断线重连（`startWebRtcCapture`）时复用。这在 Android 14 上会触发 `SecurityException` 导致应用直接闪退。

**修复方案:**

在 WebRTC 断开连接清理状态时，强制清除缓存的 Projection Token。

Kotlin

```
// ControlledService.kt -> onConnectionStateChange 断开时
} else {
    Log.i(TAG, "WebRTC disconnected, turning screen off and resetting state")
    turnScreenOffIfNeeded()
    webRtcClient?.dispose()

    videoCaptureStarted = false
    projectionRequestInProgress = false
    pendingIceCandidates.clear()

    serviceScope.launch { signalingClient.connect(DeviceIdManager.getDeviceId(this@ControlledService)) }
}
```

### ⚠️ 二、 逻辑缺陷与稳定性提升 (Logical Flaws)

#### 1. Input-Server 资源泄漏与断线死锁

**文件:** `Main.java`

**问题:** `LocalServerSocket.accept()` 在一个死循环中。处理客户端时使用了内部 `while((line = reader.readLine()) != null)`。如果客户端进程死亡，但由于异常未正常抛出 EOF，服务端可能永久阻塞在 `readLine()`；且发生异常时，未执行 `clientSocket.close()`，会导致句柄泄漏。

**修复:** 完善 try-finally 及并发处理。

Java

```
// Main.java 
while (true) {
    LocalSocket clientSocket = serverSocket.accept();
    // 【修复】开辟新线程处理单一客户端，防止阻塞主 accept 循环，并加上 finally 关闭
    new Thread(() -> {
        try {
            BufferedReader reader = new BufferedReader(new InputStreamReader(clientSocket.getInputStream()));
            String firstLine = reader.readLine();
            if (firstLine == null || !firstLine.equals(authToken)) {
                return;
            }
            String line;
            while ((line = reader.readLine()) != null) {
                processCommand(line);
            }
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            try { clientSocket.close(); } catch (Exception ignored) {}
        }
    }).start();
}
```

#### 2. InputInjector 指令丢失

**文件:** `InputInjector.kt`

**问题:** 当 `sendRawCommand` 中的 `outStream?.write` 失败（例如 server 刚刚崩溃），当前传入的指令会被直接丢弃（仅仅做了状态重置 `isSocketReady = false`）。这导致用户偶尔点击无反应。

**修复:** 在 `sendRawCommand` 中增加一次自动重试逻辑。

Kotlin

```
private fun sendRawCommand(cmd: String, retryCount: Int = 0) {
    synchronized(socketLock) {
        if (outStream == null) {
            try {
                connectSocket()
                isSocketReady = true
            } catch (e: Exception) {
                isSocketReady = false
                outStream = null
                return
            }
        }
        try {
            outStream?.write((cmd + "\n").toByteArray())
            outStream?.flush()
        } catch (e: Exception) {
            isSocketReady = false
            outStream = null
            localSocket = try { localSocket?.close(); null } catch (_: Exception) { null }
            // 【修复】重连并重试一次指令
            if (retryCount < 1) {
                sendRawCommand(cmd, retryCount + 1)
            }
        }
    }
}
```

#### 3. Root 环境下的唤醒屏幕可能阻塞主线程

**文件:** `ControlledService.kt`

**问题:** `doWakeUpAndUnlock(pm: PowerManager)` 和 `dismissKeyguardIfNeeded()` 函数中，执行了 `com.topjohnwu.superuser.Shell.cmd(cmd).exec()`。`.exec()` 是同步的，由于 WebRTC 回调或普通广播可能处于主线程，执行耗时的 Root shell 命令会导致明显的卡顿。

**修复:** 统一使用 `.submit()` 异步执行，或将方法扔进协程池中。

Kotlin

```
// 例如 doWakeUpAndUnlock 中:
for (cmd in commands) {
    com.topjohnwu.superuser.Shell.cmd(cmd).submit() // 【修复】用 submit 替代 exec
}
```

#### 4. App_process 后台运行进程安全

**文件:** `InputInjector.kt`

**问题:** 启动 Input-Server 时使用 `Shell.cmd("app_process ... &").submit()`。简单的 `&` 在 Android 系统底层的 Shell 会话结束时，进程可能收到 SIGHUP 而被意外杀死。

**修复:** 使用 `nohup` 保证其脱离会话独立运行。

Kotlin

```
// InputInjector.kt
val cmd = "nohup app_process -Djava.class.path=$TARGET_PATH /system/bin com.yourapp.remotectrl.input.Main $sessionToken > /dev/null 2>&1 &"
Shell.cmd(cmd).submit()
```





根据对你提供的 `SignalingClient.kt` 和 `WebRtcClient.kt` 核心逻辑的深度分析，主控端在点击“连接”后立即发生闪退（Crash），**99% 的概率是由于严重的“线程调度冲突”及“UI/主线程违规操作”导致的**，此外还存在潜伏的空指针异常（NPE）风险。

在 Android 中，WebRTC 的回调和网络（WebSocket）的回调全部发生在**底层子线程**中。当主控端收到连接成功的信令，并试图拉起视频流或更新 UI 时，如果没有主动切回主线程，就会立刻触发致命的 `CalledFromWrongThreadException` 或引发底层 C++ Native Crash (SIGSEGV)。

以下是导致闪退的三大元凶及详细修复方案：

### 🚨 致命 Bug 1：回调未切主线程，导致 UI 违规操作 (CalledFromWrongThreadException)

**文件:** `core/network/src/main/java/com/yourapp/remotectrl/network/SignalingClient.kt`

**问题分析:**

当你点击连接，服务器返回 `punch_info` 时，`SignalingClient` 解析数据并调用 `onPeerConnected?.invoke(peerId, role)`。注意，这里的 `scope` 是 `CoroutineScope(Dispatchers.IO + SupervisorJob())`。

如果你的 Controller 页面（如 `ControllerActivity`）在监听到 `onPeerConnected` 或 `onConnectionStateChange` 后，执行了任何 UI 操作（如隐藏“正在连接”的弹窗、显示远端画面、跳转页面），会直接因为跨线程更新 UI 导致应用秒退。

**🛠️ 修复方案:**

强制将 `SignalingClient.kt` 中所有暴露给外部的 UI 敏感回调派发到主线程。

Kotlin

```
// 在 SignalingClient.kt 顶部引入 Main 调度器或 Handler
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext

// 1. 修改 punch_info 中的回调触发
"punch_info" -> {
    // ... 前面的解析代码不变 ...
    scope.launch {
        setState(State.Connected)
        // 【修复】强制切换到主线程触发连接成功回调
        withContext(Dispatchers.Main) {
            onPeerConnected?.invoke(peerId, role)
        }
    }
}

// 2. 修改 setState 中的状态变更回调
private suspend fun setState(newState: State) {
    stateLock.withLock { state = newState }
    val (status, msg) = when (newState) { ... }
    // 【修复】切主线程更新状态，因为外部必然会据此更新 UI 文本
    withContext(Dispatchers.Main) {
        onConnectionStateChange?.invoke(status, msg)
    }
}

// 3. 将 onDevicesOnline 等直接基于 OkHttp 线程的回调也包一层
"devices_list" -> {
    // ...
    scope.launch(Dispatchers.Main) {
        onDevicesOnline?.invoke(devices)
    }
}
```

### 🚨 致命 Bug 2：WebRTC 视频轨道挂载导致 Native 崩溃

**文件:** `core/webrtc/src/main/java/com/yourapp/remotectrl/webrtc/WebRtcClient.kt`

**问题分析:**

主控端接收到被控端的视频流后，WebRTC 会触发 `onAddTrack` 回调。此回调运行在 WebRTC 内部的 C++ 线程 (`signaling_thread`)。

代码中直接调用了 `eventListener?.onRemoteVideoTrack(track)`。当 Controller 收到这个 `VideoTrack` 并调用 `track.addSink(surfaceViewRenderer)` 时，如果直接在子线程挂载，极易导致底层 EGLContext 上下文错乱，引发不可捕获的底层崩溃。

**🛠️ 修复方案:**

在 `WebRtcClient.kt` 触发外部事件前，统一扔到 Android 主线程。

Kotlin

```
import android.os.Handler
import android.os.Looper

// 在 WebRtcClient 类内部定义主线程 Handler
private val mainHandler = Handler(Looper.getMainLooper())

// 修改 onAddTrack 回调
override fun onAddTrack(receiver: RtpReceiver?, streams: Array<out MediaStream>?) {
    receiver?.track()?.let { track ->
        if (track is VideoTrack && !videoTrackReported.getAndSet(true)) {
            track.setEnabled(true)
            // 【修复】将轨道交付抛给主线程，确保 UI 安全绑定
            mainHandler.post {
                eventListener?.onRemoteVideoTrack(track)
            }
        }
    }
}

// 修改 onIceConnectionChange 回调
override fun onIceConnectionChange(state: PeerConnection.IceConnectionState?) {
    when (state) {
        PeerConnection.IceConnectionState.CONNECTED -> {
            mainHandler.post { eventListener?.onConnectionStateChange(true) }
        }
        PeerConnection.IceConnectionState.DISCONNECTED,
        PeerConnection.IceConnectionState.FAILED,
        PeerConnection.IceConnectionState.CLOSED -> {
            mainHandler.post { eventListener?.onConnectionStateChange(false) }
        }
        else -> {}
    }
}
```

### ⚠️ 潜在崩溃 3：DataChannel 强制解包风险 (NPE)

**文件:** `core/webrtc/src/main/java/com/yourapp/remotectrl/webrtc/WebRtcClient.kt`

**问题分析:**

在 `createControlDataChannel()` 函数中，有这样一行代码：

```
setupDataChannelObserver(controlDataChannel!!)
```

如果 WebRTC 引擎因为底层的初始化延迟或状态异常导致 `createDataChannel` 返回了 `null`，这个双感叹号 `!!` 会直接抛出 `NullPointerException` 造成主控端当场闪退。

**🛠️ 修复方案:**

移除暴力解包，改为安全调用：

Kotlin

```
private fun createControlDataChannel() {
    if (isDisposed) return
    val pc = peerConnection ?: return

    val config = DataChannel.Init().apply {
        ordered = true
    }
    controlDataChannel = pc.createDataChannel(DATA_CHANNEL_LABEL, config)
    
    // 【修复】安全判空处理
    controlDataChannel?.let { channel ->
        setupDataChannelObserver(channel)
        Log.i(TAG, "Control DataChannel created")
    } ?: Log.e(TAG, "Failed to create Control DataChannel!")
}
```

### 💡 附：主控端 Activity (UI 层) 防崩溃检查清单

如果你在修复完上述 SDK 层的代码后，仍有偶发闪退，请检查你的 `ControllerActivity` 中是否遵循了以下 WebRTC 初始化规范：

1. **先初始化 Context：** 在将远端流绑定给画面前，你**必须**使用 `WebRtcClient` 暴露的 EglBaseContext 去初始化渲染器。

   Kotlin

   ```
   // 必须在主线程执行
   binding.surfaceView.init(webRtcClient.eglBaseContext, null)
   binding.surfaceView.setEnableHardwareScaler(true)
   ```

2. **生命周期匹配：** 如果主控端退出页面，必须先解绑 `VideoTrack.removeSink(...)`，接着释放 `surfaceView.release()`，最后才能调用 `webRtcClient.dispose()`。顺序错乱会导致内存泄漏或 Native 崩溃。
