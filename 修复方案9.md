通过深度审查你提供的完整项目代码，我发现了导致“频繁断开且长时间离线”**以及**“幽灵设备（被控端已离线但主控端仍显示在线）”的四大深层逻辑漏洞。

这不仅仅是简单的网络波动问题，而是涉及到协程心跳自杀机制、数据列表脏读合并、协程作用域内存泄漏，以及服务端 TCP 僵尸连接（Half-open）等深水区 BUG。

以下是根本原因的万字级深度排查与完美的修复方案，请按顺序修改四个文件。

### 🚨 致命元凶一：心跳检测的“自杀式”断网（导致频繁掉线且连不上）

**🔍 原因分析：**

在 `SignalingClient.kt` 的 `startAppHeartbeat` 中，你启动了一个协程每 20 秒发送一次心跳 JSON。但是！你写了这样的错误逻辑：

Kotlin

```
val sent = ws.send(ping.toString())
if (!sent) { break } // 发送失败直接 break 退出循环！
```

WebSocket 在网络轻微抖动或底层发送队列满时，`send()` 偶尔会返回 `false`。原代码遇到一次 `false` 就直接 `break` 退出了心跳循环！**这意味着该客户端再也不会发送 Ping 了。** 随后，`healthCheckJob` 会在 45 秒后因为没收到 Pong，强行判定连接死亡并踢掉连接。这让原本没断的网络被代码硬生生“杀”死了。

**🛠️ 修复方案：移除 `break`，允许重试**

打开 `core/network/src/main/java/com/yourapp/remotectrl/network/SignalingClient.kt`，找到 `startAppHeartbeat` 方法，将 `break` 删除：

Kotlin

```
    private fun startAppHeartbeat(ws: WebSocket) {
        heartbeatJob?.cancel()
        heartbeatJob = scope.launch {
            while (isActive && !isDestroyed) {
                delay(20_000)
                try {
                    val ping = JSONObject().apply { put("type", "ping") }
                    val sent = ws.send(ping.toString())
                    if (!sent) {
                        Log.w(TAG, "Heartbeat send returned false, waiting for next cycle")
                        // 【修复1】：坚决不能 break！只记录日志，等待下一次循环自动恢复
                    }
                } catch (e: Exception) {
                    Log.w(TAG, "Heartbeat error: ${e.message}")
                    // 【修复1】：坚决不能 break！
                }
            }
        }
    }
```

### 🚨 致命元凶二：客户端列表“脏读合并”（导致主控端显示幽灵设备在线）

**🔍 原因分析：**

在 `SignalingClient.kt` 中，`devices_list`（获取全量列表）和 `device_online`（单个设备上线）**共用了同一个回调 `onDevicesOnline`**。

在 `ControllerService.kt` 接收到数据后，它只会往 `onlineDevices` 这个 Map 里 **添加 (put)** 新设备，或者移除发来 offline 消息的设备。

**致命漏洞**：当网络重连，主控端向服务端拉取全量列表 `devices_list` 时，服务端发来的是当前真正活着的设备。但是，主控端并没有**清空 (clear)** 旧的 Map！导致之前遗留在 Map 里、实际上已经断网死掉的设备永远残留在界面上，变成了“幽灵在线”。

**🛠️ 修复方案：分离“全量刷新”与“单点更新”回调**

**第一步：修改 `SignalingClient.kt`**

将原本单一的回调拆分为两个，找到文件顶部的声明，修改为：

Kotlin

```
    var onConnectionStateChange: ((String, String) -> Unit)? = null
    // 【修复2】：拆分回调，一个用于全量覆盖，一个用于单点更新
    var onDevicesRefresh: ((List<OnlineDevice>) -> Unit)? = null 
    var onDeviceUpdate: ((OnlineDevice) -> Unit)? = null
```

然后往下找到 `onWsMessage` 内部，修改这两个消息的分发逻辑：

Kotlin

```
            "devices_list" -> {
                // ... (前面解析 devices 的代码保持不变) ...
                Log.i(TAG, "Received devices_list: ${devices.size} devices")
                scope.launch(Dispatchers.Main) {
                    onDevicesRefresh?.invoke(devices) // 【修复2】：触发全量刷新
                }
            }
            "device_online", "device_offline" -> {
                // ... (前面解析 device 的代码保持不变) ...
                Log.i(TAG, "Device ${if (device.isOnline) "online" else "offline"}: ${device.id}")
                scope.launch(Dispatchers.Main) {
                    onDeviceUpdate?.invoke(device) // 【修复2】：触发单点更新
                }
            }
```

**第二步：修改 `ControllerService.kt`**

找到 `setupSignalingCallbacks()`，将旧的 `onDevicesOnline` 彻底替换为以下代码：

Kotlin

```
        // 【修复2】：全量刷新时，必须先清空本地残留的幽灵设备！
        signalingClient?.onDevicesRefresh = { devices ->
            Log.i(TAG, "onDevicesRefresh: received ${devices.size} devices")
            onlineDevices.clear() // 核心逻辑：清空脏数据！
            for (device in devices) {
                if (device.isOnline) {
                    onlineDevices[device.id] = device
                }
            }
            updateAndBroadcastDevices()
        }

        // 【修复2】：单点更新时，增删 Map
        signalingClient?.onDeviceUpdate = { device ->
            if (device.isOnline) {
                onlineDevices[device.id] = device
            } else {
                onlineDevices.remove(device.id)
            }
            updateAndBroadcastDevices()
        }
```

并在同文件中补充这个辅助方法：

Kotlin

```
    private fun updateAndBroadcastDevices() {
        val devicesCopy = onlineDevices.toMap()
        Handler(Looper.getMainLooper()).post {
            try { onDevicesUpdate?.invoke(devicesCopy) } catch (e: Exception) {}
        }
        try { sendDevicesBroadcast(devicesCopy) } catch (e: Exception) {}
    }
```

### 🚨 致命元凶三：协程作用域内存泄漏导致死锁

**🔍 原因分析：**

在 `Watchdog` 中，如果连接彻底失败，你会调用 `signalingClient.disconnect()` 然后 `new` 一个新的。

但是，在 `SignalingClient.kt` 的 `disconnect()` 方法里，你并没有调用 `scope.cancel()`。这会导致旧对象的网络接收器协程变成了僵尸协程，疯狂占用内存并引发内部锁死，导致后续实例长达几分钟连不上网。

**🛠️ 修复方案：**

在 `SignalingClient.kt` 底部找到 `disconnect()`，加上取消作用域：

Kotlin

```
    fun disconnect() {
        Log.i(TAG, "disconnect() called")
        isDestroyed = true
        wsGeneration++
        reconnectJob?.cancel()
        heartbeatJob?.cancel()
        healthCheckJob?.cancel()
        
        val currentWs = ws
        ws = null
        if (currentWs != null) {
            try { currentWs.close(1000, "disconnect") } catch (_: Exception) {}
        }
        // 【修复3】：彻底杀死协程作用域，防止泄漏造成的长时死锁
        scope.cancel() 
    }
```

### 🚨 致命元凶四：Go 服务端缺乏读超时踢人机制（TCP Half-Open 僵尸连接）

**🔍 原因分析：**

如果被控端手机突然断网（如进电梯、路由器拔电）或者被用户划掉后台，它来不及向服务端发送 WebSocket 的 Close 帧。

在 Go 服务端，这种叫 **TCP 半开连接（Half-Open）**。如果 Go 服务端不强制设置 `SetReadDeadline`，服务端会傻傻地把这个设备挂在 `s.clients` 内存里长达 15 到 30 分钟！这就是产生“幽灵设备”的最终极原因：**服务端自己都不知道客户端已经死了。**

**🛠️ 修复方案：在你的 Go 服务端（接收客户端 JSON Ping 的地方）加入倒计时踢人**

由于你客户端是每 20 秒发一次 `{"type":"ping"}`，服务端必须要求每个客户端至少在 60 秒内发来一次消息，否则强行断开并清理。

在你未展示的 `server/main.go` 中，处理 WebSocket `ReadMessage` 循环的地方（通常在 `client.readPump()` 之类的方法里），你需要加入以下逻辑：

Go

```
// 【修复4服务端】：在处理客户端消息的 for 循环中添加 ReadDeadline
const pongWait = 60 * time.Second

func (c *Client) readPump() {
    defer func() {
        c.server.removeClient(c) // 确保连接死掉时移除 s.clients 广播下线
        c.conn.Close()
    }()

    // 初始化时设置一次死线
    c.conn.SetReadDeadline(time.Now().Add(pongWait))

    for {
        _, message, err := c.conn.ReadMessage()
        if err != nil {
            break // 读超时或连接断开，自动跳出循环并踢人
        }
        
        // 每次收到任何消息（包括心跳），都续命 60 秒
        c.conn.SetReadDeadline(time.Now().Add(pongWait))

        // 解析并 dispatch
        var env Envelope
        if json.Unmarshal(message, &env) == nil {
            if env.Type == "ping" {
                // 必须回应 pong，否则客户端 45 秒后会自杀
                c.Send(Envelope{Type: "pong"})
            } else {
                c.server.dispatch(c, &env)
            }
        }
    }
}
```
